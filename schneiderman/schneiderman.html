Shneiderman's Golden Rules

Reflection on GoBloom and the Sketchbook

Golden Rules Reflection

Which rule feels most absent from your micro-project?

Rule 4: design dialogs to yield closure. GoBloom has no moment that feels finished. A round ends and the screen resets. The game ends and you get a score. Neither of those things feels like a landing. Shneiderman says sequences need a beginning, a middle, and an end, and that the end should actually communicate something. Right now the end of a round just feels like an interruption. The transition screen helped a little. But the game still does not give you a moment to breathe and register that something concluded. You are always mid-process, even when it is over.

Which rule would most improve your sketchbook interface?

Rule 8: reduce short-term memory load. That is the root problem with the landing page and it is so obvious once you name it. Sixteen unlabeled links is a direct memory burden. Every time someone visits, they have to reconstruct the sketchbook from scratch because nothing on the page holds the structure for them. Shneiderman says users should not have to remember information from one display to use it on another. My sketchbook asks you to remember what Number 4 was from a previous visit just to decide whether to click it again. One line of description per link would fix most of that. It is such a small change relative to the problem it solves.

Which rule is hardest to implement honestly?

Rule 6: permit easy reversal of actions. For the exact reason we already talked about. In GoBloom, reversal would collapse the game. The tension is that you cannot take a click back. Shneiderman says reversibility relieves anxiety and encourages exploration. But GoBloom is built on anxiety. That is the point. Removing it would make the game feel safe in a way that makes it feel pointless. So this rule and my design are in direct conflict and I think my design is right. But I cannot implement the rule without breaking what I made. That is not a failure. It is a tradeoff. Knowing the difference matters.

Did any rule contradict how you designed your system originally?

Rule 3 and rule 7 both did, and neither contradiction was intentional. Rule 3 says every user action should have informative feedback. In the original version of GoBloom, clicking a seed and getting a weed was feedback in the visual sense but not in any informative sense. Something happened. The system never told you what it meant for where you stood in the game. Rule 7 says users should feel in control and should not be surprised. The random outcome is a surprise every single time. I designed it that way because I thought unpredictability was interesting, and I still think it is. But Shneiderman would say I designed the surprise without designing the context that makes it feel fair rather than broken. The instructions screen was an attempt to fix that. Telling the player upfront that the outcome is random reframes it as a rule instead of a malfunction. That gap, between a system that keeps you in control and one that just keeps you guessing, is exactly where I needed to do more work.
